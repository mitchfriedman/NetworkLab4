The data for this lab is being processed entirely on the Server side. In the future, I will do the same thing. This is because it will make the responses that I give to the frontend much smaller since I am getting rid of all the useless data that I won't need. This is a good strategy to build a scalable and highly available API since each request is smaller. Eventually, when I implement a cache, storing less data in the cache will be useful as well. :)



In terms of preventing SQL injections I found that MySQLdb has support for named parameters that will automatically filter the strings and make them safe for using directly in queries. As a part of PEP 249 many libraries support this, including the one we are using. So I made sure to correctly use tuples in my formatted strings in the queries as outlined here.
